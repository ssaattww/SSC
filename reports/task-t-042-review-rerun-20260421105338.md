## 指摘事項

- 指摘要約または「指摘なし」:
  - Medium:
    - `src/SSC/ParallelDynamicAccessExtensions.cs:264-283,311-336` は、nested member が materialize 済み member node に見つからない場合に `materializedNode: null` へフォールバックし、そのまま `GetState()` で runtime reflection を再実行します。これにより、基底型 / interface 型で宣言された member の先にある runtime-derived property については、`GetState` が依然として getter を呼び直します。設計書は dynamic value-path `GetState` が compare 時に materialize した member state を参照し、getter を再実行しないと明記している (`doc/design/detail/02-PublicApi.md:194-199`, `doc/design/detail/09-ValueStateBehavior.md:44-46`) ため、このケースでは実装が設計契約を満たしていません。追加テスト `Compare_DynamicProjection_NestedValuePath_AllowsRuntimeDerivedMemberAccess` (`tests/SSC.E2E.Tests/ContainerAndSelectManyE2ETests.cs:365-399`) は access 継続しか確認しておらず、derived getter の副作用 / 例外が `GetState` 中に再発しないことを検証していません。runtime-derived member も non-invasive にしたいなら compare 時 materialization の扱いを広げる必要があり、そこまで行わないなら設計書の保証範囲を明示的に狭める必要があります。

## リスク

- 未解決のリスクまたは後続対応:
  - polymorphic な runtime-derived property に副作用 getter や例外 getter があると、`GetState` が依然として非侵襲ではなく、設計書どおりの timing guarantee を提供できない。
  - このまま取り込むと、dynamic value-path `GetState` の non-invasive 契約が「declared member に限る」のか「runtime-derived member も含む」のか不明確なまま残る。
  - fix では、runtime-derived member に対する side-effect / exception getter の回帰テストを追加し、実装か設計のどちらを正とするかを明確にする必要がある。
