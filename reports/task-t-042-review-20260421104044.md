## 指摘事項

- High:
  - `src/SSC/ParallelDynamicAccessExtensions.cs:63-66` で materialize 済み member node を常に優先するようにした結果、dynamic nested value path の `GetState` だけでなく `Property[modelIndex]` まで `DynamicParallelNodeView.TryGetIndex` (`src/SSC/ParallelDynamicAccessExtensions.cs:107-116`) と `ParallelNode.GetValue` (`src/SSC/ParallelNode.cs:103-106`) 経由へ切り替わっています。さらに member node 自体は `BuildMemberSlots` (`src/SSC/ParallelCompareApi.cs:133-174`) が compare 時に getter を先行実行し、例外を `ReflectionMetadataBuildFailed` として吸収して構築します。設計では T-042 の対象を dynamic value-path `GetState` のみに限定し、`Property[modelIndex]` は対象外と明記している (`doc/design/detail/02-PublicApi.md:184-199`) ので、今回の変更は公開挙動を task scope より広く変えています。未アクセスの nested property でも `Compare()` が `Issues` / `HasError` を増やし得る点まで含めて回帰で、追加テスト (`tests/SSC.E2E.Tests/ContainerAndSelectManyE2ETests.cs:243-319`) ではこの indexer 側の仕様逸脱を捕捉できていません。
- Medium:
  - `src/SSC/ParallelCompareApi.cs:125-127` は member node を declared property type で構築し、`TypeMetadataResolver.GetComparableProperties` (`src/SSC/Internal/TypeMetadataResolver.cs:11-18`) も declared type 上の public property だけを列挙しています。一方、変更前の dynamic value path 解決は runtime object に対して `current.GetType().GetProperty(...)` (`src/SSC/ParallelDynamicAccessExtensions.cs:284-294`) を使っていました。つまり、`Base Detail` / `IDetail Detail` のように基底型・interface 型で宣言された property に derived instance を入れているモデルでは、runtime 側にしかない nested member へ `root.Items[0].Detail.Label` のように辿れなくなります。polymorphic な dynamic nested access を壊すので、member node 経路へ寄せるなら runtime type との整合を保つ必要があります。
