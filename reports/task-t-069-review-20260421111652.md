## 指摘事項

- 指摘要約または「指摘なし」:
  - `Medium` `TraceLog` が未設定でも、hot path の各 `context.Trace(...)` 呼び出しで `params (string Key, object? Value)[]` が毎回生成されます。null ガードはメソッド内部 (`src/SSC/ParallelCompareApi.cs:668-686`) にあるため、既定設定の比較でも node/property/model ごとに不要な割り当てが走ります。特にループ内の呼び出し (`src/SSC/ParallelCompareApi.cs:114-130`, `178-189`, `275-280`, `314-342`, `635-647`) は入力サイズに比例して GC 負荷を増やすので、trace off が既定の API に性能退行を持ち込みます。
  - `Medium` タスク定義と設計/README は `List` / `Array` / `IEnumerable` / `Dictionary` の trace 契約を追加していますが、追加 E2E は `List` と `IEnumerable` の 2 ケースしかありません (`tests/SSC.E2E.Tests/CompareApiE2ETests.cs:107-175`)。`Array` は `GetContainerCategory` 分岐、`Dictionary` は別系統の実装 (`src/SSC/ParallelCompareApi.cs:139-252`, `525-549`) なので、このままだと今回 public に広げた trace 契約の半分が回帰検知できません。参照: `tasks/tasks-status.md:10-18`, `reports/task-t-069-implementation-20260421111613.md:8-10`, `doc/design/detail/02-PublicApi.md:276-291`, `doc/design/detail/03-ContainerRules.md:26-67`, `README.md:68-88`。
  - `Low` `tasks/tasks-status.md:7-19` の T-069 はまだ `Status: 設計・TDD 着手` のままで、実装レポートが示す「コード追加・テスト実行済み」の状態 (`reports/task-t-069-implementation-20260421111613.md:22-32`) と一致していません。tracking を入力に次工程を選ぶ skill 群がある repo なので、このままだと review 後の progress sync や完了判定を誤ります。

## リスク

- 未解決のリスクまたは後続対応:
  - `TraceLog` 無効時の割り当てを抑えないまま出荷すると、既定経路の compare 性能が入力規模に応じて悪化する。
  - `Array` / `Dictionary` trace の回帰テストが無いままだと、README / 設計で公開した出力契約を将来の変更で壊しても検知できない。
  - tracking 不整合を放置すると、task 完了判定や次 skill の選択に誤差が残る。
