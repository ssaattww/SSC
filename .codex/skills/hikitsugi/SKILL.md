# Hikitsugi

## Purpose

新しいチャットに移っても追加確認なしで作業再開できるよう、背景・経緯・決定事項・未解決事項・次アクションを欠落なく整理したハンドオーバー文書を作成する。

## Trigger

以下の依頼が来たらこのスキルを使う。
- 「新しいチャット用に引き継ぎを作って」
- 「次スレッドで再開できるメモを作って」
- 「要約ではなく前提を省略せず残して」

## Output Contract

- 目的は要約ではなく、次チャット単体で再開できる再現性の高い引き継ぎ。
- ファイル名はユーザー指定を最優先。
- ファイル名未指定時の既定:
  - `report/chat-handover-for-new-thread-YYYYMMDD_HHMMSS.md`
- 本文は必ず次の 8 章をこの順序で含める。
  - 【1. このチャットの目的】
  - 【2. 新しいチャットに切り替える理由】
  - 【3. 背景・前提条件】
  - 【4. ここまでの経緯】
  - 【5. 決定事項】
  - 【6. 未解決事項・保留事項】
  - 【7. 次のチャットで最初に依頼すべき内容】
  - 【8. 引き継ぎ本文】

## Workflow

1. 現在状態を収集する。
   - `git status --short --branch`
   - `git log --oneline -n <必要件数>`
   - 関連する `tasks/*.md`, `reports/`, 主要設定ファイル
2. 会話で確定した方針を抽出する。
   - 設計方針、命名、CI/CD、レビュー運用、未解決課題
3. 時系列で実施履歴を整理する。
   - 何を確認し、何を変更し、どのコミット/レポートで裏付けられるか
4. 未解決事項を次作業に直結する単位で明確化する。
   - 判断待ち
   - 外部依存（secret, 権限, CI状態）
   - 未push/未反映差分
5. 【7】に「そのまま貼れる依頼文」を作る。
   - 次チャットで最初の実行手順が明確であること
6. 【8】に完成版の引き継ぎ本文を作る。
   - 追加確認なしで着手できる粒度まで具体化する

## Quality Gate

- 重要情報を「不要かもしれない」で省略しない。
- 曖昧語（例: 適宜、必要に応じて）を避け、対象ファイル・ブランチ・コミットを具体化する。
- 「決定事項」と「未解決事項」を混在させない。
- 次チャットで最初に実行するコマンド/作業順が明示されている。
- `tasks/tasks-status.md`, `tasks/phases-status.md`, `tasks/feedback-points.md` の更新が必要なら同一作業で反映する。

## Do Not

- 章立てを崩さない。
- 箇条書きだけで文脈が欠ける構成にしない。
- 前提ファイルを「参照して」で済ませない。
- 既存の未解決課題を勝手に解決済みにしない。
